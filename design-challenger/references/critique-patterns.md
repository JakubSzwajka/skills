# Critique Patterns

Use these patterns to keep the review sharp, honest, and useful.

## 1. Diagnose before prescribing

Do not jump straight to redesign ideas. First identify:
- what this screen is trying to do
- who it is for
- what is getting in the way
- what evidence in the artifact supports that diagnosis

## 2. Ask hard questions first

Use questions to expose weak logic before suggesting solutions.

Examples:
- Why should the user care immediately?
- What exactly do you want them to do next?
- Why is this element visually louder than the real decision point?
- What objection is still unanswered?
- Why does this feel trustworthy enough to act on?
- What is this screen trying to say in one sentence, and does the UI support that?

## 3. Separate symptoms from root causes

Do not stop at surface comments like:
- needs more spacing
- too busy
- CTA is weak
- feels off

Translate symptoms into causes:
- the hierarchy fails because too many elements compete for first attention
- the CTA feels weak because value and urgency are not established first
- the layout feels busy because grouping is unclear and content density is unmanaged

## 4. Prioritize leverage

Do not produce laundry lists. Rank issues by likely impact on:
- comprehension
- trust
- conversion
- effort to fix

Prefer a short list of high-leverage changes over an exhaustive minor audit.

## 5. Be concrete, not decorative

Avoid vague advice like:
- make it cleaner
- modernize it
- improve UX
- add more polish

Prefer:
- reduce competing emphasis in the hero so the CTA becomes the obvious next step
- move proof closer to the action because the user is asked to commit before trust is established
- simplify the first screen to one message and one action

## 6. Use blunt language when the work is weak

Do not fake enthusiasm. If the screen is confused, say it is confused. If it looks untrustworthy, say so. Brutality is acceptable when it improves the signal.

Keep the critique professional, not insulting.

## 7. Adapt the output to the ask

Common modes:
- quick teardown
- brutal review with questions first
- category-based UX audit
- conversion-focused critique
- PRD challenge from a UX/UI angle

If the user asks for a lighter review, compress. If they ask to be grilled, intensify the questioning.

## 8. Only implement when explicitly asked

By default, stay in critique and direction-setting mode.

If asked to implement, keep the same diagnosis-first mindset, then produce directional changes. For PRD assistance, HTML reference directions are allowed, but avoid pixel-perfect specs unless specifically requested.
