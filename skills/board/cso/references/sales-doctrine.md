# Sales Doctrine

Global CSO doctrine applies across products. Product-specific claims, ICP, pricing, legal posture, and evidence stay in each repo.

## Operating Beliefs

- Sales is a learning loop, not a personality trait.
- A product without customer evidence is a hypothesis with a UI.
- One specific buyer beats a broad audience.
- A repeated painful workflow beats generic interest.
- Activation and retention beat signup volume.
- Proof assets beat abstract positioning.
- Objections are data: segment, offer, trust, copy, onboarding, product, or price.
- More building is not progress when the missing input is buyer contact.

## Good Early Sales Looks Like

- A buyer can name the current workflow and pain.
- The product replaces or reduces an existing behavior.
- The user reaches first value with visible effort and friction recorded.
- The user returns, asks for the next output, shares it, pays, refers, or gives a clear refusal.
- The founder can explain who is not a fit.

## Bad Early Sales Looks Like

- "This could be useful for everyone."
- "People said cool idea."
- Broad posting before one narrow use case works.
- Feature work justified by imagined users.
- Hidden founder labor making the product look better than it is.
- Avoiding price, payment, procurement, trust, or urgency questions.

## Builder Avoidance Check

Before recommending more build work, ask:

1. Which buyer conversation exposed this need?
2. Which activation, retention, or payment step is blocked?
3. What evidence would tell us this build was unnecessary?
4. Can the same learning happen faster through manual concierge, mockup, or direct conversation?

If there is no answer, the next action is probably sales discovery, not engineering.
