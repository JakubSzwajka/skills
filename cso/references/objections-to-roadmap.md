# Objections To Roadmap

Objections are not all product requests. Classify before acting.

## Objection Types

- **Segment mismatch:** The person is not the buyer.
- **Pain mismatch:** The problem is real but not urgent.
- **Category confusion:** They do not understand what the product is.
- **Trust gap:** They doubt reliability, privacy, data, AI, quality, or support.
- **Activation gap:** They understand but cannot reach first value.
- **Proof gap:** They want examples, references, demos, or concrete outcomes.
- **Pricing gap:** They do not see enough value for the price or plan shape.
- **Procurement gap:** They need team, invoice, compliance, approval, or enterprise features.
- **Product gap:** The core workflow cannot work without a missing capability.

## Response Policy

- Segment mismatch -> change target list, not product.
- Pain mismatch -> change positioning or segment.
- Category confusion -> improve proof asset and first explanation.
- Trust gap -> add evidence, limitations, source visibility, legal/privacy clarity, or manual review.
- Activation gap -> simplify setup, onboarding, default templates, or guided first run.
- Proof gap -> build examples before launch channels.
- Pricing gap -> test value narrative before discounting.
- Procurement gap -> usually defer unless the project intentionally moves upmarket.
- Product gap -> create a task only when several high-fit prospects hit the same missing capability.

## Task Creation Rule

Create product/build work from an objection only when:

1. the prospect is high-fit,
2. the objection blocks activation, retention, payment, or trust,
3. the same pattern appears more than once or is strategically decisive,
4. the smallest fix is clear,
5. the expected sales-learning outcome is named.

Otherwise, log it and keep selling.
